Autenticação e organization_id em API Routes

Padrão universal de todas as rotas autenticadas. Copie para suas rotas de WhatsApp.

// app/api/whatsapp/channels/route.ts — padrão de todas as rotas
import { z } from 'zod'
import { createClient } from '@/lib/supabase/server'

function json<T>(body: T, status = 200): Response {
  return new Response(JSON.stringify(body), {
    status,
    headers: { 'content-type': 'application/json; charset=utf-8' },
  })
}

export async function GET() {
  const supabase = await createClient()

  // 1. Autenticar
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return json({ error: 'Unauthorized' }, 401)

  // 2. Pegar organization_id do perfil
  const { data: me } = await supabase
    .from('profiles')
    .select('id, role, organization_id')
    .eq('id', user.id)
    .single()

  if (!me?.organization_id) return json({ error: 'Profile not found' }, 404)

  // 3. Sempre filtrar por organization_id
  const { data, error } = await supabase
    .from('whatsapp_channels')
    .select('*')
    .eq('organization_id', me.organization_id)

  if (error) return json({ error: error.message }, 500)
  return json({ data })
}

Optimistic Update ao enviar mensagem

// Padrão: setQueryData para feedback instantâneo
// Regra do copilot-instructions: preferir setQueryData sobre invalidateQueries

const sendMessage = useMutation({
  mutationFn: async (payload: SendMessagePayload) => {
    const res = await fetch(`/api/whatsapp/conversations/${conversationId}/messages`, {
      method: 'POST',
      body: JSON.stringify(payload),
    })
    return res.json()
  },
  onMutate: async (payload) => {
    // Cancelar refetch em andamento
    await queryClient.cancelQueries({ queryKey: whatsappKeys.messages.list(conversationId) })

    // Inserir mensagem otimista no cache
    const optimisticMessage = {
      id: `temp-${Date.now()}`,
      content: payload.content,
      direction: 'OUTBOUND',
      delivery_status: 'PENDING',
      created_at: new Date().toISOString(),
    }
    queryClient.setQueryData(
      whatsappKeys.messages.list(conversationId),
      (old: Message[] = []) => [...old, optimisticMessage]
    )
    return { optimisticMessage }
  },
  onError: (_err, _vars, context) => {
    // Rollback em caso de erro
    queryClient.setQueryData(
      whatsappKeys.messages.list(conversationId),
      (old: Message[] = []) => old.filter(m => m.id !== context?.optimisticMessage.id)
    )
    addToast('Erro ao enviar mensagem', 'error')
  },
  onSuccess: (data, _vars, context) => {
    // Substituir temp pela mensagem real
    queryClient.setQueryData(
      whatsappKeys.messages.list(conversationId),
      (old: Message[] = []) => old.map(m => m.id === context?.optimisticMessage.id ? data : m)
    )
  },
})

Validação de assinatura do Webhook

import { createHmac } from 'crypto'
import twilio from 'twilio'

// Twilio
function validateTwilioSignature(req: Request, rawBody: string): boolean {
  return twilio.validateRequest(
    process.env.TWILIO_AUTH_TOKEN!,
    req.headers.get('X-Twilio-Signature') ?? '',
    process.env.NEXT_PUBLIC_APP_URL + '/api/whatsapp/webhook',
    JSON.parse(rawBody)
  )
}

// Meta Cloud API Direta
function validateMetaSignature(req: Request, rawBody: string): boolean {
  const signature = req.headers.get('x-hub-signature-256') ?? ''
  const expected = 'sha256=' + createHmac('sha256', process.env.META_APP_SECRET!)
    .update(rawBody).digest('hex')
  return signature === expected
}

Integrar o WhatsApp oficial dentro do Levante CRM para que os clientes gerenciem toda a comunicação sem sair da plataforma.

  • Enviar e receber mensagens WhatsApp diretamente pelo CRM
  • Gerenciar templates de mensagem (marketing, utilidade) com aprovação da Meta
  • Enviar mensagens em massa (broadcast) para listas de contatos
  • Ver histórico completo de conversas no cockpit do deal
  • Continuar usando WhatsApp no celular normalmente (coexistência simultânea)

Princípio: zero fricção para o cliente

⛔ Sem APIs não-oficiais

4.2 Fluxos da integração

Onboarding — Conectar Canal
Cliente clica "Conectar WhatsApp" nas configurações
Popup Embedded Signup da Meta abre — cliente autoriza
Callback retorna WABA ID + Phone Number
Frontend envia para POST /api/whatsapp/channels
API registra sender no BSP + configura webhook apontando para /api/whatsapp/webhook
Canal CONNECTED ✅ — ativar flag whatsapp_integration em ai_feature_flags
Mensagem Recebida — processamento assíncrono obrigatório
Cliente WhatsApp → Meta → BSP → POST /api/whatsapp/webhook
Validar assinatura → retornar 200 imediatamente (<100ms)
Processar em background: identificar org (createStaticAdminClient) → UPSERT conversation
Auto-linkar contact por phone normalizado → criar se não encontrar
Inserir message (INBOUND) → se mídia: baixar + salvar no Supabase Storage
Supabase Realtime notifica UI → TanStack Query invalida cache
🔴 Webhook DEVE responder 200 antes de qualquer processamento
Download de mídia pode levar vários segundos. Se não responder em 5s, o BSP retenta e cria mensagens duplicadas. Responda 200 primeiro, processe em background (ctx.waitUntil() no Vercel Edge ou Promise sem await).
Mensagem Enviada
Vendedor digita → POST /api/whatsapp/conversations/[id]/messages
Janela aberta → envia texto livre (grátis)   Janela fechada → só template (cobrado)
Optimistic update no cache (setQueryData) → feedback instantâneo
Chama BSP → Insere message (OUTBOUND) com delivery_status = 'PENDING'
Webhook de status → SENT → DELIVERED → READ (só progressão, nunca retroceder)

Como funciona
Cloud API suporta coexistência nativa desde 2024. Vendedor continua usando o celular normalmente — sem QR code, sem desconectar. Mensagens de ambos os lados são sincronizadas via webhook.
📱 → CRM

Mensagens enviadas pelo celular chegam no CRM via webhook. Marcar sent_from_phone = true se o BSP fornecer esse metadata.

💻 → Celular

Mensagens enviadas pelo CRM aparecem no celular automaticamente. Sincronização bidirecional nativa.

⚠ Verificar antes de implementar o badge
O campo sent_from_phone depende de metadata no payload do webhook do BSP. Confirmar se está disponível no BSP escolhido antes de construir o badge visual.

Criar lib/query/whatsapp.ts com whatsappKeys. Ver padrão de código na seção acima.

Não criar WhatsApp Context. O padrão do CRM é TanStack Query hooks — o CRMContext existente é só fachada por compatibilidade. Criar hooks customizados em lib/query/whatsapp.ts e usá-los nos componentes diretamente.
DEALS_VIEW_KEY: Para invalidar qualquer query relacionada ao cockpit de deal, usar queryClient.invalidateQueries({ queryKey: DEALS_VIEW_KEY }) importado de @/lib/query.

  • Regra do copilot-instructions: preferir setQueryData sobre invalidateQueries quando seguro
